# Чек-лист перед mock interview

Использовать перед каждым mock, а не как список для зубрёжки. Отмечать только доказанное. Если пункт не закрыт, выбрать один конкретный короткий опыт из силлабуса, а не читать всё заново.

## Обязательный минимум

- [ ] Могу объяснить, что Go компилируется, чем module отличается от package и как запустить тест.
- [ ] На примере предсказываю aliasing slice, nil map и value/pointer receiver.
- [ ] Читаю wrapped error через `errors.Is`/`errors.As` и объясняю, где уместен `panic`.
- [ ] Выбираю interface или generic с причиной, а не «всегда interface».
- [ ] Запускаю table-driven test и понимаю результат benchmark/fuzz.
- [ ] Останавливаю goroutines через `context`, знаю владельца закрытия channel и проверяю `-race`.
- [ ] Объясняю, почему mutex, channel и atomic не взаимозаменяемы.
- [ ] Пишу HTTP handler с валидацией, timeout, JSON error и тестом; закрываю response body клиента.
- [ ] Выполняю SQL в транзакции, объясняю MVCC/lock/deadlock и читаю ключевые строки `EXPLAIN`.
- [ ] Называю source of truth при cache-aside, последствия Pub/Sub без подписчика и природу Kafka offset.
- [ ] Выбираю at-least-once с идемпотентностью, backoff и DLQ вместо выдуманного exactly-once.
- [ ] Защищаю outbox, называю границы saga/CDC и случай, когда оставить монолит.
- [ ] Читаю Dockerfile/Compose/Nginx/Kubernetes manifest: health, timeout, retries, probes, limits.

## Формат ответа на собеседовании

1. Сначала сформулировать условия задачи и риск.
2. Предложить решение и один разумный альтернативный вариант.
3. Назвать компромисс: latency, consistency, стоимость, сложность или операционный риск.
4. Сказать, как проверить результат: тест, метрика, лог, запрос, профилирование или нагрузочный сценарий.
5. При неизвестном не угадывать: обозначить предположение и короткий способ проверки.
