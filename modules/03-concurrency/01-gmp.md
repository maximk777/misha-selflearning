# GMP

> Практика: `labs/concurrency/gmp/LAB.md`, проверка: `labs/concurrency/gmp/CHECK.md`

G — goroutine, M — OS-thread, P — ресурс выполнения Go-кода. Scheduler сопоставляет G с M через P; `GOMAXPROCS` ограничивает число P. Preemption не гарантирует порядок: не строим корректность на порядке печати. Прогноз: почему 100 goroutine не означают 100 потоков? Наблюдай harness командами `go -C labs/concurrency/gmp/starter run . -jobs 4 -work 1000000 -gomaxprocs 1` и с `-gomaxprocs 2`, затем перескажи модель.

## Где это применяется в реальном backend

1. **Много фоновых задач** — scheduler мультиплексирует G на M/P; число goroutines не равно параллелизму.
2. **CPU-bound расчёт** — `GOMAXPROCS` ограничивает одновременное исполнение Go-кода; больше G не ускоряет работу бесконечно.
3. **Блокирующий syscall** — runtime может отделить заблокированный M; зависимость от порядка запуска остаётся ошибкой.

## Глубокое погружение

P хранит ресурсы выполнения, M исполняет G, queues и work stealing балансируют runnable G; это концептуальная модель, детали runtime меняются. Инвариант correctness не зависит от scheduling order. Costs: goroutine stacks, context switches, queue contention. Long CPU loops исторически влияли на fairness, но preemption не является timing guarantee. Наблюдай уже предоставленный scheduler-пример при `GOMAXPROCS=1/2`, записывая только различия нескольких запусков и не принимая порядок логов или время одного запуска за гарантию.

## Мини-проект

### Результат

На готовом harness в `labs/concurrency/gmp/starter` проведи наблюдательный эксперимент: меняй только flags `-gomaxprocs`, `-jobs`, `-work`, затем опиши различия нескольких запусков. Самостоятельно создавать goroutines или runner в этой теме не требуется; внутренний WaitGroup не нужно понимать или менять.

### Разрешённые знания

Весь Go Core и модель GMP; goroutines разрешены только внутри предоставленного примера, без самостоятельной реализации, channels/context/HTTP.

### Проверка

Из корня: `go -C labs/concurrency/gmp/starter run . -jobs 4 -work 1000000 -gomaxprocs 1`, затем та же команда с `-gomaxprocs 2`. Каждую выполни минимум три раза и запиши наблюдения без требования одинакового порядка.

### Критерии приёмки

- [ ] корректность не зависит от порядка вывода;
- [ ] эксперимент не добавляет собственную конкурентную архитектуру;
- [ ] ученик объясняет роли G/M/P и предел параллелизма.

### Усложнение после первой версии

Изменить объём CPU-цикла в предоставленном примере и сформулировать новую проверяемую гипотезу о scheduler.
