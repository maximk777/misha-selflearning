# atomic

Typed atomics (`atomic.Int64`, `atomic.Bool`) подходят для одного независимого значения. CAS: загрузить старое, проверить, `CompareAndSwap`; повторить при конфликте. Если несколько полей образуют один инвариант, используй mutex: набор отдельных atomic не становится транзакцией.

## Где это применяется в реальном backend

1. **Независимый счётчик** — `atomic.Int64` считает завершения; согласованный snapshot нескольких полей требует Mutex.
2. **Флаг остановки** — atomic bool публикует состояние; context лучше для cancellation дерева и ожидания.
3. **Однократный переход** — CAS выбирает одного победителя; retry loop под contention расходует CPU.

## Глубокое погружение

Typed atomic операции неделимы и имеют memory-order guarantees Go; не смешивай atomic/non-atomic access к одной переменной. CAS сравнивает старое состояние и меняет его только при совпадении, возможен цикл конфликтов. ABA и multi-field invariants не исчезают. Alignment безопаснее с typed wrappers, копировать их после использования нельзя. Докажи `-race`, high-contention test и benchmark CAS против mutex; отсутствие race не доказывает бизнес-correctness.

## Мини-проект

### Результат

Добавь runner обязательный typed-atomic counter завершённых jobs; после полного запуска его значение должно точно равняться числу принятых jobs. Составная статистика остаётся под mutex.

### Разрешённые знания

Предыдущие темы, typed atomics и CAS.

### Проверка

Из корня: `cd project/concurrency-runner && go test -race -count=50 ./...`; параллельный test принимает известное число jobs и проверяет exact total counter после ожидания завершения.

### Критерии приёмки

- [ ] atomic counter имеет exact total, равный числу принятых и завершённых jobs;
- [ ] нет смешанного доступа;
- [ ] ученик объясняет, почему CAS не транзакция.

### Усложнение после первой версии

Сравнить CAS loop и Mutex benchmark при низком и высоком contention.
