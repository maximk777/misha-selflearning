# CAP

При network partition распределённая система выбирает availability или consistency для конкретной операции; «CA всегда» не существует при реальном partition. Уточняй инвариант: можно ли показать stale баланс, потерять заказ или принять двойную оплату?

## Где это применяется в реальном backend

1. **Резервирование товара между двумя регионами** — CAP заставляет явно решить, можно ли принять заказ при потере связи с владельцем остатка. Availability сохраняет приём заказов, но допускает oversell; строгий запрет oversell означает отказ части операций.
2. **Чтение каталога из реплики** — stale-ответ часто приемлем ради доступности и низкой latency. Для баланса, лимита или статуса оплаты та же модель опасна: пользователь может действовать по уже неверным данным.
3. **Выбор лидера и запись заказа** — quorum защищает единственность принятого состояния во время partition. Цена — недоступность меньшинства и рост latency; CAP не выбирает настройки вместо бизнес-инварианта.

## Глубокое погружение

Partition — потеря или неопределённая задержка сообщений, поэтому отличить «узел умер» от «сеть медленная» нельзя без допущений. Инвариант формулируется на уровне операции: linearizable write может требовать quorum, а stale read — нет. Стоимость consistency — дополнительные round trips, координация и отказ операций; стоимость availability — конфликты и последующее reconciliation. Edge cases: split brain, асимметричная связь, clock skew, повтор запроса после timeout. Production failure проявляется не словом CAP, а измеримо: двойное резервирование, stale age, rejected writes, recovery time. Доказательство — таблица сценариев и эксперимент с разрывом сети, где сравниваются ответы, latency и итоговое состояние обеих сторон.

## Мини-проект

### Результат

Бизнес-сценарий: магазин принимает заказы из двух регионов и должен осознанно пережить потерю связи между ними. В существующем `labs/architecture/01-cap/starter/` дополни `scenarios.md` ADR и создай маленький исполняемый симулятор двух реплик: для `GET status` и `POST reserve` выбери поведение при partition, затем покажи минимум три измеримых сценария — нормальную связь, разрыв и восстановление. Не получай готовую схему у наставника: сначала предложи модель состояния и ожидаемые исходы; не подключай внешнюю БД или новую инфраструктуру.

### Разрешённые знания

Go, tests, goroutines/channels/context, HTTP, PostgreSQL/Redis/Kafka из пройденных модулей и текущая тема CAP. Новая инфраструктура не требуется: partition можно моделировать флагом или тестовым transport.

### Проверка

После создания `go.mod` и теста запусти из корня репозитория `cd labs/architecture/01-cap/starter && go test ./... -count=1`; отдельно выведи таблицу `scenario / HTTP result / latency / state A / state B`. Намеренно задержи связь дольше deadline и сравни прогноз с результатом.

### Критерии приёмки

- [ ] ADR называет бизнес-инвариант и разные решения для read и write, а не выбирает «CP/AP систему целиком»;
- [ ] happy path, partition и recovery воспроизводимы тестами без потери или скрытого изменения результата;
- [ ] измерены хотя бы rejected writes, stale reads или время восстановления;
- [ ] Миша объясняет, какой trade-off принят и почему timeout не доказывает смерть узла.

### Усложнение после первой версии

Добавь асимметричный partition: A видит B, а B не видит A, и дополни ADR наблюдаемым failure mode.
