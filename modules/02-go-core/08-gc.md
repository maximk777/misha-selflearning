## Результат за 5–15 минут

Запусти memory starter несколько раз: `go -C labs/go/09-memory-gc/starter run .`. Он печатает `Alloc` до и после явного `runtime.GC`; числа не являются стабильным benchmark.

## Модель

GC освобождает недостижимую память, но не закрывает files, sockets и database rows — ими владеет явный cleanup. Паузы и allocations зависят от нагрузки, поэтому смотреть надо профиль и метрики настоящего процесса.

## Прогноз → опыт → поломка

**Один вопрос:** гарантирует ли `runtime.GC()` одинаковый `Alloc` во всех запусках? Проверь несколько запусков и объясни, почему нет.

## Пересказ и сдача

**Один вопрос:** какой ресурс GC не вернёт вместо тебя? Этот deep dive не нужен для перехода к HTTP.

## Где это применяется в реальном backend

1. **Пиковый batch** — allocations повышают heap и частоту GC; ручной `runtime.GC` в request path обычно ухудшает latency.
2. **Кэш ссылок** — достижимые объекты не освобождаются, даже если бизнесу не нужны; нужна политика eviction/ownership.
3. **Файлы и соединения** — GC управляет памятью, не своевременным close внешних ресурсов.

## Глубокое погружение

Concurrent tracing collector ищет достижимые объекты; точные runtime phases и tuning меняются. Инвариант: пока существует ссылка, память жива. Costs выражаются CPU, heap target и короткими паузами; высокая allocation rate создаёт давление. Edge cases — retention через slices/maps, finalizers без гарантии времени, misleading `Alloc` одного запуска. Докажи `runtime.MemStats`, benchmark и heap profile серией запусков, не единичным числом.

## Мини-проект

### Результат

Продолжи `project/order-report/domain`: создай воспроизводимый memory-сценарий, где большой batch становится недостижимым после удаления удерживающей ссылки.

### Разрешённые знания

Предыдущие темы, GC, reachability, benchmarks/memory stats.

### Проверка

Из корня репозитория: `go -C project/order-report test ./...`; диагностический запуск `go -C project/order-report run .` печатает память до/после, несколько повторов фиксируются как диапазон.

### Критерии приёмки

- [ ] вывод не обещает точное освобождение по одному числу;
- [ ] внешний ресурс закрывается явно;
- [ ] ученик объясняет reachability и retention.

### Усложнение после первой версии
