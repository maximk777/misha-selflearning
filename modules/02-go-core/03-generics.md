## Модель

Generic параметризует алгоритм типом и constraints проверяются при компиляции. Interface задаёт поведение и может хранить разные runtime values. Не заменяй interface generic-ом автоматически: спроси, нужен ли общий алгоритм или полиморфное поведение.

## Где это применяется в реальном backend

1. **Поиск элемента по ID** — generic убирает дублирование алгоритма для доменных типов; constraint не должен обещать лишние операции.
2. **Копирование/фильтрация slices** — тип результата сохраняется compile time; reflection здесь обычно сложнее.
3. **Поведение хранилища** — interface уместнее, если различается runtime-поведение; generic repository часто скрывает разные запросы и инварианты.

## Глубокое погружение

Type set constraint определяет разрешённые операции; `any` не разрешает `+`, `comparable` нужен для `==` и map key. Inference сокращает вызов, но ошибки типов остаются compile time. Нулевая длина требует явного `(zero, false)` или error. Возможная стоимость зависит от compiler и кода, поэтому сейчас не делаем заявлений о скорости. Докажи компиляционными negative cases и обычными tests для нескольких типов.

### Разрешённые знания

Предыдущие темы, generics, constraints, slices/maps; без concurrency и reflection.

### Проверка

Из корня репозитория: `go -C project/order-report test ./...` для двух типов, empty input и missing value.

### Критерии приёмки

- [ ] generic обобщает алгоритм, а не бизнес-поведение;
- [ ] constraint минимален;
- [ ] ученик объясняет zero value и выбор против interface.
