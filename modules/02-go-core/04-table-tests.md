# Table-driven tests

> Уровень: `core`
> Время: 60 минут
> Предпосылки: functions
> Практика: `labs/go/08-tests-bench-fuzz/LAB.md`
> Экзамен: `exams/go-core.md`, GC-5

## Результат за 5–15 минут

Выполни `go -C labs/go/08-tests-bench-fuzz/starter test ./...`. Один `TestAdd` запускает named subtests для нескольких входов.

## Модель

Table-driven test отделяет данные cases от одинакового arrange/act/assert. Имя subtest делает failure понятным и позволяет запустить один case. Тест должен проверять поведение, а не внутреннюю реализацию.

## Прогноз → опыт → поломка

**Один вопрос:** какой case упадёт, если `Add` всегда возвращает `0`? Временно сделай такую поломку, посмотри имя failure и почини.

## Пересказ и сдача

**Один вопрос:** почему два cases в одном test не обязаны быть двумя отдельными функциями? Покажи output и объясни имя subtest.

## Где это применяется в реальном backend

1. **Матрица скидок** — одна проверка покрывает пороги и роли клиента; без имён failure не объясняет бизнес-case.
2. **Переходы статусов** — cases фиксируют разрешённые и запрещённые пары; тест внутреннего helper мешает refactor.
3. **Ошибка ввода** — таблица отделяет expected category от текста; mutable fixtures между cases создают зависимость порядка.

## Глубокое погружение

Каждый subtest должен иметь независимый arrange и наблюдаемый assert. Map как таблица добавляет недетерминированный порядок; slice делает порядок явным. `t.Run` создаёт scope имени; `t.Parallel` требует особой осторожности с shared state и здесь не нужен. Costs — время и диагностика: слишком широкая таблица скрывает разные сценарии. Докажи выборочным `go -C project/order-report test -run 'TestName/case' -count=1 ./...` и намеренной поломкой одного case.

## Мини-проект

### Результат

Продолжи `project/order-report/domain`: покрой таблицами расчёт суммы, ошибки и переходы статусов.

### Разрешённые знания

Предыдущие темы и package `testing`: table-driven tests и named subtests.

### Проверка

Из корня репозитория: `go -C project/order-report test ./...`; затем `go -C project/order-report test -run 'TestName/case' -count=1 ./...` для одного subtest.

### Критерии приёмки

- [ ] happy path, boundaries и error paths имеют говорящие имена;
- [ ] cases не делят изменяемое состояние;
- [ ] assertions проверяют контракт, не реализацию.

### Усложнение после первой версии

Добавить regression case для самостоятельно найденной поломки.
