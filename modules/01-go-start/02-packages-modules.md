## Результат за 5–15 минут

В `labs/go/01-syntax/starter` выполни `go env GOMOD` и `go test ./...`. Первая команда показывает активный `go.mod`, вторая запускает tests всех packages под текущим module.

## Модель

`package` — единица компиляции и namespace исходников. `module` описан `go.mod`: это граница версий и зависимостей. Один module может содержать несколько packages. В JavaScript ближе всего package к папке с импортируемым кодом, но Go package не выбирается по произвольному runtime resolver.

## Где это применяется в реальном backend

1. **Разделение сервиса** — `main` запускает приложение, отдельный package считает отчёт; экспортировать нужно только публичный контракт.
2. **Общие зависимости** — `go.mod` фиксирует module path и версии; случайный второй `go.mod` меняет границу сборки.
3. **CI-проверка** — `go test ./...` обходит packages модуля; запуск из неверной директории способен пропустить код или не найти module.

## Глубокое погружение

Package компилируется как единица из файлов одной директории, import path начинается от module path. Импорт не должен образовывать цикл. Имена с заглавной буквы экспортируются, что увеличивает API и связанность. Стоимость лишних packages — навигация и зависимости, а монолитного package — неясное ownership. Доказательство: `go list ./...`, `go env GOMOD`, `go test ./...`.

## Мини-проект

### Результат

Раздели `order-report` на запускаемый package и package расчёта, сохрани прежний CLI-вывод.

### Разрешённые знания

Синтаксис предыдущей темы, packages, modules, import/export, `go run` и `go test`.

### Проверка

Из корня репозитория: `cd project/order-report && go env GOMOD && go list ./... && go test ./... && go run .`.

### Критерии приёмки

- [ ] один `go.mod` задаёт понятную границу;
- [ ] package расчёта не импортирует запускаемый package;
- [ ] ученик объясняет module path, package name и экспортируемые имена.

### Усложнение после первой версии

Добавить второй запускаемый package, использующий тот же расчёт для краткого отчёта.
