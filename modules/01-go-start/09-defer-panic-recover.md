## Модель

`error` — обычное значение, которое caller обязан рассмотреть. `defer` выполняется при выходе из функции и удобен для cleanup. `panic` — не замена ожидаемой ошибки; `recover` уместен на редкой границе процесса/worker, чтобы вернуть контролируемую ошибку или лог.

## Где это применяется в реальном backend

1. **Ошибочный заказ** — ожидаемая валидация возвращает `error`; panic не должен завершать весь batch из-за пользовательских данных.
2. **Cleanup отчёта** — `defer` закрывает открытый ресурс на каждом return path; defer после неуспешного открытия вызовет неверный cleanup.
3. **Граница background worker** — `recover` может не дать panic убить процесс, но обязан записать причину; внутри бизнес-логики он скрывает дефект.

## Глубокое погружение

Deferred calls выполняются LIFO; аргументы вычисляются в момент `defer`, named results могут быть изменены deferred closure. Panic раскручивает stack и запускает defers; recover работает только внутри deferred вызова той же goroutine. Ownership ресурса возникает после успешного acquire. Production failures — swallowed error, double close, panic без stack. Докажи простыми `go test` cases для порядка return/defer, `errors.Is` и локального recovery с сохранённой причиной; стоимость defer будет измеряться позже, а не угадываться сейчас.

### Критерии приёмки

- [ ] expected errors не превращены в panic;
- [ ] cleanup происходит ровно один раз после успешного acquire;
