# net/http

`Handler` получает `ResponseWriter` и `*Request`; router может быть `http.ServeMux`. Сервер обязан иметь read/write/idle timeouts. Handler сначала валидирует method/path/input, затем пишет headers/status/body. Один handler не должен тайно владеть запуском process.

## Где это применяется в реальном backend

1. **Приём задачи через API** — `POST /tasks` превращает request в команду приложения; handler отвечает за HTTP-границу, а не за запуск процесса или хранение данных.
2. **Чтение состояния задачи** — `GET /tasks/{id}` согласует method, path, status и body; ответ `200` для любого исхода лишает клиента возможности отличить отсутствие задачи от успеха.
3. **Health endpoint** — `GET /health` помогает балансировщику увидеть живой процесс; глубокая проверка всех зависимостей может сама перегрузить сервис и не заменяет liveness.

## Глубокое погружение

`ServeMux` выбирает handler, после чего requests обслуживаются независимо. `ResponseWriter` задаёт однонаправленный переход: после записи status или тела заголовки уже нельзя надёжно изменить. Инварианты: валидировать до side effect, записывать один ответ, не хранить request-scoped состояние глобально. Стоимость дают соединения, goroutines, буферы и медленный клиент; опасны неограниченное тело, двойной `WriteHeader`, panic и отсутствующие timeouts. Поведение доказывается `httptest`, проверкой method/status/header/body и конкурентным тестом с `go test -race`.

## Мини-проект

### Результат

Сделай HTTP-границу Task API: клиент создаёт задачу и запрашивает её по id, а ошибки имеют наблюдаемые HTTP-статусы. До кода опиши ownership request, handler и in-memory данных.

### Разрешённые знания

`net/http`, функции, структуры, interfaces, maps, ошибки и ранее пройденная concurrency. Внешний router, PostgreSQL и middleware не требуются.

### Проверка

Выполни `curl` для happy path, неизвестного id и неверного method; затем `go test ./...` и `go test -race ./...`.

### Критерии приёмки

- [ ] создание и чтение задачи дают согласованные status и body;
- [ ] неверный method/input и отсутствующая задача различимы;
- [ ] handler не запускает процесс и не пишет два ответа;
- [ ] объяснены ownership response и цена неограниченного body.

### Усложнение после первой версии

Добавить предел размера входного тела и доказать тестом отказ для превышения.
