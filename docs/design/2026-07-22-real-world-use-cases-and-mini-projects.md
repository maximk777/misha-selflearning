# Реальные use cases и мини-проекты для каждой темы

Дата: 22 июля 2026 года.

## Цель

Каждый урок должен отвечать на два практических вопроса: где эта тема встречается в реальном Go backend и что Миша может самостоятельно сделать с уже доступными знаниями. Теория без прикладного контекста не считается полноценным уроком, но мини-проект не должен забегать вперёд по программе.

## Область изменений

Изменить:

- все тематические файлы `modules/*/*.md`, кроме обзорных `README.md`;
- `templates/TOPIC.md` и при необходимости `templates/LAB.md`;
- tutor skills `misha-lesson`, `misha-review`, `misha-exam`, `misha-mock-interview` и `misha-progress`;
- новый repo-local skill для автоконспектов;
- новый repo-local skill для аудита и дополнения учебных тем;
- проверяющий скрипт курса, чтобы обязательные практические блоки не исчезли при следующих правках.

Не изменять решения в `solutions/` и не отмечать прогресс ученика по факту обновления курса.

## Структура каждой темы

В каждом тематическом файле должны появиться два самостоятельных блока.

### Где это применяется в реальном backend

Блок содержит не меньше трёх конкретных ситуаций. Для каждой ситуации нужно назвать:

- продуктовую или эксплуатационную задачу;
- роль изучаемого механизма;
- границу применимости или типичную ошибку.

Примеры должны быть узнаваемыми: обработка HTTP-запросов, фоновые задачи, ограничение внешнего API, транзакции заказов, кэширование каталога, повторная доставка событий, graceful shutdown, диагностика latency. Общие формулировки вроде «используется в высоконагруженных системах» без объяснения не подходят.

### Мини-проект

Блок задаёт самостоятельную работу и содержит:

1. Наблюдаемый результат.
2. Бизнес-сценарий.
3. Ограничения задачи без готовой архитектуры и кода решения.
4. Разрешённые знания: текущая тема и перечисленные пройденные темы.
5. Команды или способ проверки.
6. Критерии приёмки: happy path, ошибки, тесты и объяснение решения.
7. Одно усложнение, которое наставник выдаёт только после первой рабочей версии.

Мини-проект может расширять уже начатый учебный проект, если это уменьшает искусственность задачи. Он не обязан создавать отдельную директорию на каждую тему: тематический файл задаёт контракт, а реализация может жить в существующей лаборатории или накопительном проекте.

## Правило доступных знаний

Источником порядка тем служит `roadmap/SYLLABUS.md`. Обязательная реализация использует только:

- текущую тему;
- темы, стоящие раньше неё в syllabus;
- базовые инструменты, явно указанные в предпосылках темы.

Будущие технологии разрешено упоминать только в блоке реальных применений с пометкой, что механизм будет изучен позже. Их нельзя требовать в коде, командах проверки или критериях приёмки.

Примеры границы:

- в уроке про functions допустим CLI-анализатор списка цен, но не HTTP handler и не PostgreSQL;
- в уроке про goroutines допустима конкурентная обработка функций с искусственной задержкой, но ещё нельзя требовать channels до соответствующего урока;
- после изучения HTTP client и concurrency допустим HTTP dispatcher с worker pool, semaphore, `context`, timeout и сбором ошибок;
- Kafka-проект может использовать PostgreSQL только потому, что PostgreSQL уже пройден раньше по syllabus.

Если production-use case требует будущих знаний, наставник объясняет связь, но выбирает для самостоятельной реализации более раннюю модель того же ограничения.

## Глубина урока

Быстрый результат за 5–15 минут остаётся входом в тему, но не заменяет глубокое понимание. После первого наблюдаемого результата каждый обязательный урок раскрывает:

- внутреннее устройство механизма;
- инварианты и ownership;
- стоимость по памяти, CPU, блокировкам или сетевым ресурсам;
- граничные случаи и типичные production-сбои;
- способ доказать поведение тестом, race detector, benchmark, profile, логом или метрикой;
- вопросы уровня собеседования: «почему так», «что происходит внутри» и «какой trade-off».

Например, тема slices не заканчивается на `append`: Миша должен понимать descriptor, backing array, `len/cap`, aliasing, reallocation, удержание большого массива маленьким subslice и отличие копирования slice от копирования элементов. Тема channels включает устройство ожидания send/receive, поведение buffered и unbuffered channel, ownership закрытия, блокировки, `nil`/closed channel, cancellation, утечки goroutines и связь с happens-before.

Глубина подаётся слоями: наблюдение → самостоятельная практика → внутренний механизм → production-последствия → защита решения. Наставник не вываливает весь deep dive до первой попытки, но тема не закрывается на одном работающем примере.

## Поведение наставника

`misha-lesson` должен вести мини-проект по следующему контракту:

1. Показать реальные use cases и связать их с опытом ученика.
2. Дать условия мини-проекта и попросить Мишу сначала предложить модель решения.
3. Не выдавать starter-код, архитектуру или пошаговое решение до самостоятельной попытки.
4. Давать подсказку только по прямой просьбе ученика, по одной подсказке за сообщение. Подсказка должна указывать следующий наблюдаемый шаг, а не раскрывать решение.
5. После присланного кода или вывода сначала назвать работающую часть, затем конкретные дефекты и последствия.
6. Привести маленький локальный пример принципа исправления, не переписывая проект за ученика.
7. Попросить новую версию и повторять review-цикл до выполнения критериев.
8. После рабочей первой версии выдать предусмотренное усложнение.
9. Завершить проект только после проходящих проверок и принятого объяснения trade-offs своими словами.

`misha-review` использует тот же цикл для повторной попытки. `misha-exam` может проверять защиту готового мини-проекта, но не подсказывает во время экзамена.

`misha-mock-interview` перед выбором вопросов читает `EVIDENCE.md`, `EXAMS.md` и `REVIEW_QUEUE.md`. В первую очередь он проверяет просроченные повторы, прошлые неверные модели и слабые места, затем добавляет вопросы по остальным доказанно пройденным темам. Ошибка не исчезает из приоритета после одного правильного ответа: требуется новая практическая или устная проверка в другом контексте.

Фраза «идеальное решение» означает не максимальную архитектуру, а решение без известных дефектов в согласованном scope: корректное, проверяемое, понятное и соответствующее текущему уровню курса.

## Автоконспект по пройденному материалу

Добавить skill, который срабатывает на просьбы «дай конспект», «собери конспект по channels» или «повтори всё пройденное».

Если область не указана, наставник задаёт ровно один вопрос: «По какой теме или по всем пройденным?» Если названа тема, которой ещё нет среди доказанно пройденных, наставник честно сообщает это и предлагает либо конспект текущего изученного фрагмента с явной пометкой, либо ближайшую пройденную тему.

Конспект строится из `modules/`, кода и доказательств в `progress/`, а не только из общего знания модели. Структура:

1. Короткая ментальная модель.
2. Где применяется в реальном backend.
3. Как механизм устроен внутри.
4. Ключевые инварианты и граничные случаи.
5. Ошибки, которые Миша уже допускал, с исправленной моделью.
6. Фрагменты или выводы из выполненных мини-проектов.
7. Вопросы для самопроверки без ответов.
8. Что ещё не доказано или требует повторения.

Режим «по всем пройденным» группирует материал по модулям и не включает непройденные темы как будто они изучены. Допустим короткий раздел «что дальше», явно отделённый от конспекта.

## Согласованность журнала прогресса

`misha-progress` должен поддерживать одни и те же факты во всех представлениях:

- `EVIDENCE.md` хранит сырые доказательства и принятые объяснения;
- `STATUS.md` хранит текущий этап и следующий шаг;
- `EXAMS.md` хранит вопросы, ответы, неверные модели и уровень владения;
- `REVIEW_QUEUE.md` хранит слабые места, срок и результат повторной проверки.

Нельзя писать «тема пройдена», если завершена только лабораторная, но не выполнены критерии темы. В таком случае формулировка должна разделять «практика выполнена» и «тема ожидает зачёта». Проверка должна находить очевидные противоречия: слабое место из экзамена без записи в очереди повторения, закрытая тема без доказательства и утверждение об отсутствии доказательств при непустом журнале.

## Накопительные проекты

Чтобы не получить 80 несвязанных игрушек, темы объединяются в последовательные продуктовые линии:

- Go Start: CLI-обработчик заказов и отчётов;
- Go Core: тестируемая библиотека доменных правил и диагностика производительности;
- Concurrency: планировщик фоновых задач, завершающийся worker pool и semaphore;
- HTTP: Task API и HTTP client;
- PostgreSQL: хранилище заказов с конкурентными сценариями;
- Redis: кэш и поток обновлений;
- Kafka: обработка событий с retries и idempotency;
- Architecture: проектирование надёжного потока заказа;
- Infrastructure: упаковка и доставка уже созданного сервиса;
- Microservice: финальная интеграция изученных механизмов.

Каждый урок даёт небольшой законченный результат, а последний мини-проект модуля объединяет результаты, которые Миша уже реализовывал раньше.

## Эталон для concurrency

После прохождения worker pool, semaphore, `context` и HTTP client Миша получает проект HTTP dispatcher:

- принять список URL;
- обрабатывать задания фиксированным числом workers;
- ограничить semaphore число одновременных HTTP-запросов;
- учитывать timeout и cancellation;
- собрать статусы и ошибки без data race;
- доказать лимит конкурентности тестом;
- корректно завершить goroutines.

Наставник сначала принимает самостоятельный дизайн и код. Затем проверяет ownership каналов, backpressure, cancellation, закрытие response body, утечки, race safety и тесты. Готовая реализация из `solutions/` не показывается.

## Автоматическая проверка курса

После `git pull` все существующие темы уже должны содержать готовые use cases, deep dive и мини-проекты. Ученик не запускает генерацию учебного содержания и не получает разные задания в зависимости от локальной модели.

Добавить двухслойную проверку.

Первый слой — детерминированный shell-checker. Он проходит по существующим тематическим файлам и падает, если файл не содержит обязательных заголовков, а мини-проект не содержит результата, разрешённых знаний и критериев приёмки. Он также печатает точный список неполных файлов и найденных пропусков. Checker контролирует структуру и согласованность, но не сочиняет текст автоматически: иначе формально заполненные шаблоны могут заменить реальные учебные сценарии.

Второй слой — repo-local skill сопровождения курса. По команде владельца «проверь и дополни темы» он:

1. Запускает checker.
2. Читает каждый отмеченный файл вместе с его местом в `roadmap/SYLLABUS.md`.
3. Проверяет, какие знания доступны к этому уроку.
4. Дописывает конкретные backend-use cases, глубокий слой и мини-проект.
5. Повторно запускает checker и выборочно проверяет содержательное качество.

Skill не меняет `progress/`, `solutions/` и уже выполненный учеником код. Он не затирает существующий хороший материал только ради унификации формулировок.

Ручная выборочная проверка охватывает минимум:

- раннюю тему Go Start;
- тему на границе появления новой возможности, например channels;
- накопительный concurrency-проект;
- тему с внешней системой, например PostgreSQL или Kafka;
- infrastructure-тему без требования ещё не пройденного Kubernetes.

## Критерии готовности изменений

- Во всех 80 тематических файлах есть реальные use cases и мини-проект.
- У каждого мини-проекта явно указаны разрешённые знания.
- Ни один обязательный проект не требует будущую тему по `roadmap/SYLLABUS.md`.
- Tutor skills запрещают непрошеные подсказки и задают повторяемый review-цикл.
- Автоконспект спрашивает область, использует только доказанно пройденное и включает прошлые ошибки.
- Mock interview приоритизирует слабые места и повторно проверяет их в новом контексте.
- Уроки содержат глубокий слой внутреннего устройства, граничных случаев и production-последствий.
- Concurrency включает HTTP dispatcher после прохождения необходимых тем.
- Файлы прогресса не противоречат друг другу по статусу тем, доказательствам и слабым местам.
- Структурная проверка курса и существующие проверки проходят.
- После клонирования или `git pull` весь материал доступен без локальной генерации; audit-skill нужен для будущего сопровождения курса.
- Пользовательские изменения в `progress/` сохранены без правок.
